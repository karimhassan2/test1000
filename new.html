By Anika Patel, UiPath Community MVP | April 2026

i18n Localization Studio Web Best Practices

Why Internationalization Matters for RPA

As enterprises deploy automations across global offices, handling multiple languages, date formats, currencies, and character encodings becomes critical. A bot that works perfectly with English-language invoices may fail catastrophically when processing Japanese, Arabic, or German documents. This guide covers the essential patterns for building locale-aware automations in UiPath Studio and Studio Web.

1. Unicode and Character Encoding

Always use UTF-8 encoding for all text processing. UiPath's .NET runtime handles Unicode natively, but issues arise when reading files, calling APIs, or interacting with legacy systems that use different encodings. 

// Force UTF-8 when reading files
string content = System.IO.File.ReadAllText(filePath, System.Text.Encoding.UTF8);

// When writing CSV output
System.IO.File.WriteAllText(outputPath, csvData, new System.Text.Encoding.UTF8Encoding(true));

Watch out for BOM (Byte Order Mark) issues — some systems expect it, others choke on it. The UTF8Encoding(true) constructor includes the BOM; use UTF8Encoding(false) to omit it.

2. Date and Number Formatting

Never hardcode date formats. Use CultureInfo to parse and format dates according to the target locale: 

// Parse a German date string
Dim germanCulture = New System.Globalization.CultureInfo("de-DE")
Dim invoiceDate = DateTime.Parse("15.03.2026", germanCulture)

// Format for US output
Dim usCulture = New System.Globalization.CultureInfo("en-US")
Dim formatted = invoiceDate.ToString("MM/dd/yyyy", usCulture)
// Result: "03/15/2026"

The same applies to number formatting — German uses comma as decimal separator (1.234,56) while US uses period (1,234.56). Always specify the culture when parsing numeric strings from documents.

3. Resource Files for Multi-Language Bots

Store all user-facing strings in .resx resource files rather than hardcoding them in workflows. Create separate resource files for each locale:

  • Strings.resx — Default (English)
  • Strings.de-DE.resx — German
  • Strings.ja-JP.resx — Japanese
  • Strings.ar-SA.resx — Arabic (note: RTL layout considerations)

Access strings using Properties.Resources.ResourceManager.GetString("key", culture). This pattern makes it trivial to add new languages without modifying workflow logic.

4. OCR and Document Understanding for Non-Latin Scripts

UiPath Document Understanding supports multiple languages, but accuracy varies significantly. For CJK (Chinese, Japanese, Korean) documents:

  • Use the UiPath Document OCR engine with language-specific models
  • Pre-process images to improve contrast for complex characters
  • Validate extracted text against expected patterns (e.g., Japanese postal codes: 〒XXX-XXXX)
  • Consider using Google Cloud Vision or Azure Computer Vision for languages with lower UiPath OCR accuracy

5. Right-to-Left (RTL) Language Support

Arabic and Hebrew text requires special handling. When generating reports or emails that mix RTL and LTR content: 

// Use Unicode directional markers
string rtlText = "\u200F" + arabicContent + "\u200F";  // Right-to-Left Mark
string mixedContent = englishHeader + "\n" + rtlText + "\n" + englishFooter;

For PDF generation, ensure your template engine supports bidirectional text. Libraries like iTextSharp handle this natively, but you may need to specify font files that include Arabic/Hebrew glyphs.

6. Currency and Financial Data

Always store monetary values as decimal types with explicit currency codes (ISO 4217). Never rely on locale-specific currency symbols for parsing: 

// Bad: Ambiguous
Dim amount = Decimal.Parse("$1,234.56")  // Which dollar? USD? CAD? AUD?

// Good: Explicit
Dim amount = 1234.56D
Dim currency = "USD"
Dim formatted = amount.ToString("C", New CultureInfo("en-US"))
// Result: "$1,234.56"

7. Time Zones and Scheduling

For global automation scheduling, always store timestamps in UTC and convert to local time only for display: 

// Store in UTC
Dim utcNow = DateTime.UtcNow

// Convert for display in Tokyo
Dim tokyoZone = TimeZoneInfo.FindSystemTimeZoneById("Tokyo Standard Time")
Dim tokyoTime = TimeZoneInfo.ConvertTimeFromUtc(utcNow, tokyoZone)

Be aware of daylight saving time transitions — schedule critical automations outside the 2:00-3:00 AM window in affected time zones.

8. Testing Across Locales

Create a test matrix covering your target locales. Key test scenarios:

  • Date parsing with locale-specific formats
  • Number parsing with different decimal/thousands separators
  • String sorting (locale-aware collation)
  • Character length validation (CJK characters are often wider)
  • Email templates with mixed-direction text
  • PDF generation with non-Latin fonts

9. API Integration Considerations

When calling external APIs, always specify the Accept-Language header and handle response encoding correctly: 

// Set locale headers for API calls
httpClient.DefaultRequestHeaders.Add("Accept-Language", "de-DE,de;q=0.9,en;q=0.8");
httpClient.DefaultRequestHeaders.Add("Accept-Charset", "utf-8");

Some APIs return error messages in the requested language — make sure your error handling logic can parse these regardless of locale.

10. Orchestrator Multi-Tenant Locale Configuration

In multi-tenant Orchestrator deployments, configure locale settings at the folder level. Each folder can have its own default culture, which affects how queue items display dates and numbers in the Orchestrator UI. Use the Orchestrator API to programmatically set folder-level culture: 

PUT /odata/Folders({folderId})
{
    "DisplayName": "APAC Operations",
    "Culture": "ja-JP",
    "TimeZoneId": "Tokyo Standard Time"
}

Conclusion

Internationalization isn't an afterthought — it's a fundamental design decision that affects every layer of your automation architecture. By following these patterns from the start, you'll build automations that work reliably across languages, locales, and time zones. The investment in proper i18n pays dividends as your automation program scales globally.
